# Phase D — Commerce & Member System Design
_Popper Tulimond — Approved 2026-05-04_

---

## Overview

Phase D is the full business engine. It wires checkout, member accounts, and the Initiation Night ritual into a self-running system. Once set up, it requires zero manual action from Logan month to month unless he wants to change the drop settings.

---

## Guiding Principles

- The gate is at **checkout**, not at browsing or cart-building. Non-members browse, add items, build carts freely.
- Cart persists in `localStorage` indefinitely. A Pledge who builds their cart on the 14th returns on the 15th and picks up exactly where they left off.
- Every month's drop runs automatically. Logan only logs in if he wants to adjust numbers.
- The checkout never leaves poppertulimond.com.
- Membership setup happens at peak intent — immediately after payment. Text and email are the safety net, not the primary path.

## 1. Data Model

### New Tables

**`members`**
```sql
id              uuid primary key
phone           text not null unique
email           text not null
name            text
setup_token     uuid unique
setup_token_expires_at timestamptz
passkey_registered boolean default false
member_since    timestamptz
created_at      timestamptz default now()
```
Created at the moment payment is confirmed. `setup_token` is valid for 7 days and consumed on passkey registration.

**`member_webauthn_credentials`**
```sql
id              uuid primary key
member_id       uuid references members(id)
credential_id   text not null unique
public_key      bytea not null
counter         integer not null default 0
created_at      timestamptz default now()
```
Mirrors `webauthn_credentials` for admins. Same WebAuthn infrastructure, separate table.

**`orders`**
```sql
id                        uuid primary key
member_id                 uuid references members(id) -- nullable until passkey registered
stripe_payment_intent_id  text not null unique
phone                     text not null
email                     text not null
items                     jsonb not null
  -- [{item_id, display_name, size, price_cents, quantity}]
total_cents               integer not null
created_at                timestamptz default now()
```

**`initiation_drops`**
```sql
id              uuid primary key
drop_month      date not null unique  -- always the 16th of the month
opens_at        timestamptz not null  -- midnight EST on the 16th (auto-set)
early_access_at timestamptz not null  -- 11:45pm EST on the 15th (auto-set)
closes_at       timestamptz not null  -- default: 12:29am EST on the 16th
available_count integer not null      -- Logan sets; auto-carries from last month
sold_count      integer not null default 0
is_open         boolean not null default true  -- manual kill switch
created_at      timestamptz default now()
```

Auto-carry forward: if no row exists for the current month, the system creates one by copying `available_count` and `closes_at` from the previous month's row. Logan sees it pre-populated in the admin panel.

Sellout check (atomic, race-safe):
```sql
UPDATE initiation_drops
SET sold_count = sold_count + 1
WHERE id = ? AND sold_count < available_count
RETURNING *
```
If this returns no rows, the drop is sold out — reject the purchase.

**`early_access_tokens`**
```sql
id          uuid primary key
phone       text not null
token       uuid not null unique
drop_id     uuid references initiation_drops(id)
created_at  timestamptz default now()
```
Generated by the cron job on the 15th. Embedded in the Twilio text. Grants an `early_access_session` cookie, valid from 11:45pm EST until the drop closes.

## 2. Store State Machine

Store state is **computed** from the clock + active drop row. Nothing requires manual toggling for a standard month.

| Period | Non-member sees | Member sees |
|--------|----------------|-------------|
| 17th → 11:44pm EST 15th | "Sign up for early access" | Live Vault, shop anytime |
| 11:45pm–11:59pm EST 15th | Checkout open (Pledges with link only) | Live Vault |
| Midnight–closes_at EST 16th | Checkout open for everyone | Live Vault |
| Sold out OR closes_at hit | "This month's initiation is complete. See you next month. Members shop anytime." | Live Vault |

**Protocol Gate** fires only when a non-member attempts to complete checkout — not on browsing, size selection, or add-to-cart.

**Gate logic (in order):**
1. Is the user a logged-in member? → Always pass.
2. Is the drop sold out or past `closes_at`? → Block with sold-out message.
3. Does the user have a valid `early_access_session` cookie? → Pass if current time ≥ 11:45pm on the 15th.
4. Is current time between `opens_at` and `closes_at`? → Pass.
5. Otherwise → Block with appropriate message.

## 3. Cart

- State: React context + `localStorage`. Survives tab close, browser restart.
- Non-members: Constable items only. Non-Constable items visible but grayed out ("Members only").
- Pricing rule: 1 Constable per cart at initiation price. A 2nd Constable auto-prices at member rate with an inline note.
- Size selection (S/M/L/XL/XXL) is required before "Add to Cart" activates.
- Cart persists until the user clears cache or completes checkout.

## 4. Checkout Flow

All on-site. Never redirects to Stripe-hosted pages.

**Step 1 — Cart review**
Slide-in panel (consistent with existing overlay pattern). Shows: product photo, item name, size, price. "Checkout" CTA at bottom.

**Step 2 — Checkout screen**
Full-page. Order summary at top. Payment options:
- **Apple Pay** button (shown only when browser supports Payment Request API)
- **Google Pay** button (shown only when available)
- **"Pay another way"** — expands to Stripe Payment Element (card form)

**Step 3 — Payment**
- Apple Pay / Google Pay: native OS sheet, Face ID / double-click to confirm.
- Card: fill form, submit. Stripe confirms Payment Intent client-side.

**Step 4 — Order confirmation**
On client-side payment success, call `/api/orders/confirm`:
- Creates `orders` row
- Creates `members` row with `setup_token` (UUID, 7-day expiry)
- Atomically increments `sold_count` on active drop
- Returns `setup_token`

Redirect immediately to `/membership-setup?token={setupToken}`.

Stripe webhook fires in background as the authoritative record — idempotent, handles any edge cases.

**Stripe integration:**
- Payment Intents API
- Stripe Elements (Payment Element for card, Payment Request Button for wallets)
- Webhook endpoint: `/api/stripe/webhook`
- New env vars: `STRIPE_SECRET_KEY`, `STRIPE_PUBLISHABLE_KEY`, `STRIPE_WEBHOOK_SECRET`

## 5. Membership Registration Flow

**`/membership-setup?token={setupToken}`**

Page tone: urgent, welcoming. Copy direction:
> "You're in. But your membership isn't active yet. Register now and you can shop the Vault any time — not just once a month. This takes 10 seconds."

Single CTA: **"Activate My Membership"** → WebAuthn passkey registration (same flow as admin, pointed at `members` table).

On success: token consumed, `passkey_registered` set to true, `member_since` stamped, `orders.member_id` backfilled. Redirect to Vault as logged-in member.

If token is expired or already used: show a friendly error with instructions to contact the brand.

**Failsafe sequence (if tab closed before registering):**

| Timing | Channel | Message |
|--------|---------|---------|
| ~5 min after purchase | Twilio SMS | "You're almost a member. Finish your registration: [link] — shop the Vault anytime instead of waiting until next month." |
| +1 day | Email (Resend) | Branded HTML email, same message |
| +3 days | Email (Resend) | Reminder |
| +6 days | Email (Resend) | "Your registration link expires tomorrow." |
| Day 7 | Token expires | Contact Logan for a new link |

Email templates built with React Email — fully branded to Popper Tulimond (dark background, gold type, brand voice from `context/DISPATCH_001_THE_VOID.md`).

New env vars: `RESEND_API_KEY`, `FROM_EMAIL`

## 6. Cron Jobs

Both run on Vercel Cron.

**Drop setup cron — runs 1st of each month at 9am EST**
- Checks if an `initiation_drops` row exists for this month
- If not, creates one by copying `available_count` and `closes_at` from last month
- Logan sees it in the admin panel and can adjust before the 15th

**Pledge blast cron — runs 15th of each month at 3:45pm EST**
```
45 20 15 * *   (UTC — 3:45pm EST = 20:45 UTC)
```
Note: Vercel Cron is UTC-only. This fires at 3:45pm EST (November–March) and 4:45pm EDT (March–November). The drift is acceptable — the text still arrives 7+ hours before the 11:45pm window. If exact timing matters in the future, the cron can fire at a fixed UTC time and a Next.js route handler can queue the sends via a scheduled DB job.
1. Pull all `sms_signups` where `segment = 'pledge'`
2. Generate one `early_access_token` per phone for this month's drop
3. Fire Twilio text to each: *"Tonight is New Member Initiation. Here is your early access link to shop at 11:45pm EST before everyone else. — Popper Tulimond [link]"*

**Membership reminder cron — runs daily at 10am EST**
- Finds all `members` where `passkey_registered = false` and `setup_token_expires_at` is in the future
- Sends the appropriate email based on days since `created_at` (day 1, 3, 6)

## 7. Admin Panel Additions

Two new tabs in the Edit Pages panel (owner only).

**Initiation Night tab**

Pre-drop controls:
- `available_count` — editable integer input
- `closes_at` — time picker (pre-filled with last month's value)
- Save button

Live during the drop (auto-refreshes every 30 seconds):
- "X / Y sold" progress
- Store status badge: Open / Closed / Sold Out
- Time remaining until auto-close
- Manual kill switch: "Close Store Now" / "Reopen Store"

**Members tab**
- Table: name, phone, email, member since, passkey registered (yes/no)
- "Resend Setup Link" button per row (for support cases)
- Read-only. No editing member data from this panel.
