# Persistence & Publishing System — Design Spec

**Goal:** Allow Studio Mode changes (model positions, dots, shadows, images) to be saved permanently to disk and published, with change detection to prevent accidental data loss.

**Architecture:** Extract `MODEL_INVENTORY` to `src/data/inventory.ts` as the single source of truth. A dev-only Next.js API route (`POST /api/save-inventory`) receives the current studio state, runs it through the existing `exportInventoryCode()` serializer, and overwrites the file on disk. Next.js hot-reload picks up the change instantly. Two buttons in the Studio sidebar (`SAVE CHANGES` and `PUBLISH LIVE`) drive this flow. A `beforeunload` guard prevents accidental navigation while unsaved changes exist.

**Tech Stack:** Next.js 15 App Router, TypeScript, Node.js `fs` module (server-side only), React `useEffect` + `useState`, Tailwind CSS v4, Framer Motion (button state transitions).

## 2. API Route

**File:** `src/app/api/save-inventory/route.ts`

**Node.js imports required:** `import fs from 'fs'; import path from 'path';`

### Request
```
POST /api/save-inventory
Content-Type: application/json
Body: { slots: StudioSlot[] }
```

`StudioSlot[]` is the studio's internal editing type (from `studioTypes.ts`). The API route passes it directly to `exportInventoryCode(slots)` which already handles the `StudioSlot → ModelSlot` serialization — no conversion needed in the route.

### Response
```json
{ "ok": true }               // success
{ "ok": false, "error": "…" } // failure
```

### Behavior
1. **Dev-only guard:** If `process.env.NODE_ENV !== 'development'`, return `HTTP 403` immediately.
2. Parse `slots` from request body; return 400 if missing or not an array.
3. Call `exportInventoryCode(slots)` — this returns just the `const MODEL_INVENTORY = [...]` block (not a full file).
4. Prepend a **hardcoded file boilerplate header** (see Section 5). The boilerplate is a static string in the route file — not read from disk — so it never goes stale.
5. Build the complete file content in memory as a single string.
6. Write atomically: `fs.writeFileSync(path.join(process.cwd(), 'src/data/inventory.ts'), content, 'utf-8')`.
7. Return `{ ok: true }`.

### Error handling
- Wrap both `exportInventoryCode(slots)` and `fs.writeFileSync(...)` in a try/catch. If either throws or `exportInventoryCode` returns an empty string, return `{ ok: false, error: e.message }` with HTTP 500.
- Validate `slots`: return 400 if `slots` is `undefined`, `null`, or not an array. An empty array `[]` is valid (user may have removed all characters).
- **Crash-during-write risk:** `writeFileSync` is atomic per POSIX on most systems, but a process crash during the write could truncate the file. This is acceptable risk for a dev-only tool. The mitigation is git — `inventory.ts` should be committed before making Studio changes.

## 3. Studio UI — Save & Publish Buttons

**File:** `src/components/studio/StudioInspector.tsx`

### New prop
```ts
onSave: () => Promise<void>;
```

### Button layout
Pinned to the bottom of the sidebar, above the footer padding. Two buttons stacked vertically, full width.

**SAVE CHANGES button states:**
| State   | Appearance                                                   |
|---------|--------------------------------------------------------------|
| idle    | Charcoal bg, white text                                      |
| loading | Spinner + "Saving…" — button disabled                        |
| success | Checkmark + "Saved" (2s timeout, then back to idle)          |
| error   | "✕ Save failed" amber text inline below button (3s, then idle) |

The error message from `data.error` displays as a small label directly beneath the save button — not a toast. The button itself returns to idle text while the error label is visible.

**PUBLISH LIVE button states:**
| State   | Appearance                                  |
|---------|---------------------------------------------|
| idle    | Outlined, quieter (secondary style)         |
| active  | Runs save, then shows "Preparing Production Build…" for 3s, then returns idle |

> **Note:** PUBLISH LIVE is a **UI placeholder only** in this phase. It triggers the same save flow as SAVE CHANGES, then uses a hard-coded `setTimeout(3000)` to display "Preparing Production Build…" before returning to idle. A `// TODO: wire to deploy endpoint` comment is placed at the publish logic site. No actual deployment or CI/CD integration is wired in this phase.

### Save state machine
```
idle → loading (on click)
loading → success (on API ok)
loading → error (on API error)
success → idle (after 2s timeout)
error → idle (after 3s timeout)
```

## 4. CollectionOverlay Wiring

**File:** `src/components/CollectionOverlay.tsx`

### New state
```ts
const [hasUnsavedChanges, setHasUnsavedChanges] = useState(false);
```

### Change tracking
- Set `hasUnsavedChanges(true)` any time `studioSlots` changes while in studio mode.
- Reset to `false` after a successful save.

### `handleSave` function
```ts
async function handleSave() {
  const res = await fetch('/api/save-inventory', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ slots: studioSlots }),
  });
  const data = await res.json();
  if (!data.ok) throw new Error(data.error ?? 'Save failed');
  setHasUnsavedChanges(false);
}
```

`handleSave` throws on error. `StudioInspector` calls `onSave()` inside a try/catch and manages its own button state machine (`loading → success | error`). `CollectionOverlay` does not need to catch — the error is surfaced entirely through the inspector's UI state.

Pass `onSave={handleSave}` to `<StudioInspector>`.

### beforeunload guard
```ts
useEffect(() => {
  if (!hasUnsavedChanges || !isStudioMode) return;
  const handler = (e: BeforeUnloadEvent) => {
    e.preventDefault();
    e.returnValue = '';
  };
  window.addEventListener('beforeunload', handler);
  return () => window.removeEventListener('beforeunload', handler);
}, [hasUnsavedChanges, isStudioMode]);
```

> **Behavior clarification:** This shows the browser's native "Leave site? Changes you made may not be saved." confirmation dialog. The user can still choose to leave — this is intentional. We do not hard-block navigation. The `e.returnValue = ''` line is required for cross-browser support (Chrome ignores just `e.preventDefault()`).

### `hasUnsavedChanges` prop flow
`hasUnsavedChanges` is **not** passed to `StudioInspector`. The inspector manages its own `saveState` (`'idle' | 'loading' | 'success' | 'error'`) independently. `CollectionOverlay` uses `hasUnsavedChanges` only for the `beforeunload` guard. These are two separate concerns.

## 5. File Boilerplate for inventory.ts

The API route must write a valid TypeScript file. The generated file structure:

```ts
// src/data/inventory.ts
// ⚠️ This file is auto-generated by Studio Mode. Do not edit manually.

import type { ShadowConfig } from "@/components/studio/studioTypes";

export interface OutfitItem {
  id: string;
  name: string;
  collection: string;
  colorway: string;
  price: string;
  type: "public" | "vault";
  dotPosition: string;
  lookbook?: { id: string; src: string; label?: string }[];
}

export interface ModelSlot {
  id: string;
  position: string;
  scale: string;
  mobileScale: string;
  zIndex: number;
  imageSrc: string;
  outfit: OutfitItem[];
  shadow?: ShadowConfig;
}

export const MODEL_INVENTORY: ModelSlot[] = [
  // ... generated by exportInventoryCode()
];
```

`CollectionOverlay.tsx` imports `ModelSlot`, `OutfitItem`, and `MODEL_INVENTORY` from `@/data/inventory`. Import chain: `studioTypes.ts` (no upstream deps) → `inventory.ts` (imports ShadowConfig from studioTypes) → `CollectionOverlay.tsx`. No circular dependency.

## 6. exportInventoryCode Compatibility

`exportInventoryCode(slots: StudioSlot[])` in `studioUtils.ts` handles all `StudioSlot` fields. The function output is the array body only — the API route wraps it in the full file header.

**Required modifications — full lookbook round-trip:**

The lookbook data must survive the full cycle: Studio UI edit → Save (export) → Reload (import). Two code changes are required:

**A) Export fix** — The current implementation hardcodes `lookbook: [],` for every dot (line 144 of studioUtils.ts), which erases lookbook data on save. Update to serialize `dot.lookbook` properly:

```ts
// Replace:
lines.push(`        lookbook: [],`);

// With:
if (dot.lookbook.length === 0) {
  lines.push(`        lookbook: [],`);
} else {
  lines.push(`        lookbook: [`);
  for (const item of dot.lookbook) {
    const label = item.label ? `, label: "${item.label}"` : '';
    lines.push(`          { id: "${item.id}", src: "${item.src}"${label} },`);
  }
  lines.push(`        ],`);
}
```

**B) Import fix** — `RawOutfitItem` in `studioUtils.ts` currently has no `lookbook` field, so `modelSlotToStudio` always initializes dots with `lookbook: []` even after a save that includes lookbook data. Two changes needed in `studioUtils.ts`:

1. Add `lookbook?: { id: string; src: string; label?: string }[]` to the `RawOutfitItem` interface.
2. In the dot mapping inside `modelSlotToStudio`, change `lookbook: []` to `lookbook: item.lookbook ?? []`.

This ensures that when the user reloads the Studio after a save, the lookbook items they added are restored from `inventory.ts`.

**Shadow serialization:** The existing shadow export (line 149) already outputs `offsetX/Y` as integers and `scaleX/Y`, `opacity` with decimal precision. No changes needed.

**Type definitions:** `OutfitItem` and `ModelSlot` are defined inline in the generated `inventory.ts` (see Section 5 boilerplate). They are **not** imported from `studioTypes.ts` — only `ShadowConfig` is imported. This is explicit by design: the generated file must be self-contained apart from that single import.

## 7. Security & Edge Cases

- **Production safety:** 403 if `NODE_ENV !== 'development'`. Safe to deploy.
- **Corrupt write prevention:** Build the full file string in memory before calling `writeFileSync` — no partial writes.
- **Hot-reload:** Next.js detects the file change and re-compiles automatically — no manual refresh needed after save.
- **Concurrent saves:** Studio disables the SAVE button during loading state to prevent double-submit.
- **Exit Studio without saving:** `beforeunload` fires on refresh/tab-close. Navigating away within the SPA (if any nav links existed) is not a concern currently.
- **Hot-reload timing:** The save API returns as soon as `writeFileSync` completes. Next.js's dev server file watcher detects the `inventory.ts` change and triggers a hot module replacement within ~1 second, automatically refreshing the page. The "Saved ✓" confirmation in the sidebar reflects the file write succeeding — not the HMR cycle. This is the correct behavior: the file is saved, the dev server will catch up on its own. No explicit HMR synchronization is needed.
